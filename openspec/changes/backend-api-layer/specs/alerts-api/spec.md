# Alerts API — Specification

## Purpose

Expose active alerts generated by AI Core agents, ordered by severity. Includes a real-time SSE stream for new alerts via PostgreSQL LISTEN/NOTIFY.

## Requirements

### Requirement: List active alerts sorted by severity

The system MUST expose GET /api/v1/alerts/ returning active alerts ordered by severity (critical > high > medium > low), filterable by region_id, severity, and active_only.

#### Scenario: Returns alerts sorted by severity

- GIVEN alerts exist with severity "high", "critical", and "medium"
- WHEN GET /api/v1/alerts/?region_id=cordoba_pilot
- THEN the system MUST return alerts ordered critical first, then high, then medium
- AND each item MUST include alert_id, severity, event_type, region_id, created_at, is_active, and executive_summary

### Requirement: Single alert detail

The system MUST expose GET /api/v1/alerts/{alert_id} returning full alert detail with messages by audience and recommended actions.

#### Scenario: Returns alert detail

- GIVEN alert "alert-123" exists
- WHEN GET /api/v1/alerts/alert-123
- THEN the system MUST return alert_id, severity, messages_by_audience, and recommended_actions

### Requirement: Active alerts count

The system MUST expose GET /api/v1/alerts/active/count/ returning counts grouped by severity for a region.

#### Scenario: Returns counts by severity

- GIVEN cordoba_pilot has 3 critical, 5 high, and 2 medium alerts
- WHEN GET /api/v1/alerts/active/count/?region_id=cordoba_pilot
- THEN the system MUST return { critical: 3, high: 5, medium: 2, total: 10 }

### Requirement: Acknowledge alert

The system MUST expose PATCH /api/v1/alerts/{alert_id}/acknowledge/ to mark an alert as viewed (soft state, backend only).

#### Scenario: Acknowledges alert

- GIVEN alert "alert-123" is active
- WHEN PATCH /api/v1/alerts/alert-123/acknowledge/
- THEN the system MUST return 200 and mark the alert as acknowledged

### Requirement: SSE stream for new alerts

The system MUST expose GET /api/v1/alerts/stream/ (Accept: text/event-stream) that emits new_alert events when alerts are inserted, using PostgreSQL LISTEN/NOTIFY.

#### Scenario: Streams new alert event

- GIVEN a client is connected to /alerts/stream/
- WHEN a new alert is inserted in the database
- THEN the system MUST emit an SSE event with type "new_alert" and payload containing alert_id, severity, region_id, and event_type
